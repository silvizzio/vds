Buttons carry the primary actions on a surface. On dense command centre views,
limit each panel to one primary button so operators can find the main action
without scanning. Everything else steps down to secondary, outline, or ghost.

## When to use

- Committing an action: save, apply, confirm, delete.
- Opening a modal, drawer, or panel.
- Submitting a form or advancing a step.

## When not to use

- Navigating between pages or views. Use a link instead.
- Toggling a single on/off state. Use a switch or checkbox.

## Guidance

<DoDont>
  <Do>Lead the label with a verb that names the outcome: Save changes, Delete snapshot, Apply filter.</Do>
  <Dont>Use vague labels like OK or Submit that hide what the action does.</Dont>
  <Do>Keep one primary button per panel so the main action stays obvious.</Do>
  <Dont>Stack multiple primary buttons side by side and force the operator to choose.</Dont>
</DoDont>

<Accessibility>

## Keyboard

- **Tab** moves focus to the button.
- **Enter** and **Space** activate it.

## Screen reader

- Renders a native `button`, so role and state are announced without extra ARIA.
- When the label is an icon only, provide `aria-label` with the action text.

## Focus and contrast

- Focus shows a 2px accent ring offset from the container, visible in light and dark.
- Label and container meet WCAG AA contrast (4.5:1) across all variants.
- Never signal state with color alone. Pair destructive intent with a clear label.

</Accessibility>
